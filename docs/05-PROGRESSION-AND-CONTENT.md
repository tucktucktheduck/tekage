# 05 · PROGRESSION & CONTENT

> The ladder, the profile, and where songs come from. Sandbox-first, but with a
> climb for those who want it.

## Shape: sandbox + ladder

TKG is **a sandbox instrument you just open and play** — *and* a
**Duolingo/Chess.com-style ladder** layered on top for players who want to
progress. The two coexist: free-play any unlocked content anytime; the ladder
adds goals, unlocks, and a sense of getting better.

Principles from the creator:

- Unlocking levels should be **fun**, the core compulsion loop.
- **Skip-ahead test-outs:** a skilled player can unlock a **whole group** at once
  by demonstrating skill (pass a harder checkpoint) — they don't have to grind
  the easy rungs. If they can't, they **drop back and drill**.
- It's a *ladder*, not a wall: density Versions (07) already give per-song
  difficulty; the ladder sequences songs/skills across the catalog.

## Difficulty axes

1. **Density Version** within a song: Easy·Core → Medium·Two-Voice → Hard·Full
   (notes/sec). Generated by the extractor (07).
2. **Mechanical demand:** shift frequency, octave-jump range, simultaneity,
   tempo.
3. **Speed** (assist): the SPEED slider scales tempo for practice.

## Skill trees (confirmed)

Progression is organized as **multiple skill trees**, Duolingo-style, e.g.:

- a **chords** tree,
- a **speed / moving-quickly** tree,
- an **expert** tree,

(more as the game grows). You can **skip ahead through a tree** if you can pass
its checkpoints, and **completing songs unlocks whole parts/branches at once**.
Start easy, climb to harder levels, unlock skills as you go.

A "level" = (song × Version) with a target (clear it / hit a score/accuracy).
Clearing yields **stars** (e.g. 1–3 by accuracy) → currency for unlocks.

## Test-out mechanic

- Each **group** (a themed/leveled set of songs) has a **checkpoint** — a
  representative hard chart.
- Pass the checkpoint cold → the whole group unlocks.
- Fail → the group opens rung-by-rung as you clear its members (drill path).
- This rewards real skill without gatekeeping practice.

## Profile / persistence

`ProgressStore` (see 01) owns the `Profile`:

```ts
Profile = {
  unlocks: Set<levelId>,
  stars: Map<levelId, 0..3>,
  settings: { speed, assist, skinId, mappingId, mode },
  ownedPacks: Set<packId>,        // skins/backgrounds/teklets/song packs
}
```

- **Storage is swappable** (interface, not implementation): in-memory in the
  sandbox artifact, **IndexedDB** in the real web app, **cloud account** later
  for cross-device + purchases. Build to the interface now.
- Settings persistence (speed, last skin, mapping) is the *cheapest* first win
  and should land early in the real app.

## Where songs come from (acquisition)

Today: **user loads their own MIDI/MusicXML.** Long-term problem: most people
don't have MIDI files — acquisition is itself a design challenge.

Staged answer (see 06):

- **Now:** local upload + a few bundled demo songs.
- **Next:** a **built-in starter library** (cleared/licensed pieces) so a
  non-technical player has something to play immediately.
- **Later:** **buy songs** (licensed MIDI/MusicXML packs) — the content
  monetization arm. Requires a licensing pipeline and a `SongSource` store
  adapter. Possibly community-contributed charts with rights handling.

Keep `SongSource` an interface so "upload," "library," and "store" are just
adapters behind the same `get(id) → ParsedSong`.

## Monetization tie-in

- TKG core + base songs: **free**.
- Cosmetic packs (skins/backgrounds/teklets) and **song packs**: **paid**.
- Unlock currency (stars) gates *progression*, money gates *cosmetics/content* —
  keep these separate so it never feels pay-to-win over the *game*.

## Open questions (re-ask)

- Are stars/score even present in stage 1, or does progression come later? (02)
- Group/level taxonomy: by genre, by mechanic (shifting drills, jump drills), by
  difficulty tier — or a mix? (Duolingo-style skill tree vs. Chess.com rating?)
- Does the ladder live per-song (earn the busier Version) **and** across the
  catalog, or just across? (Creator leaned: picking density may be enough
  within a song; ladder across songs.)

## Acceptance criteria (when progression is built)

- [ ] `ProgressStore` interface with in-memory + IndexedDB adapters; logic never
      touches storage directly.
- [ ] Level = (song × Version); clearing writes stars via `ProgressStore`.
- [ ] Test-out: passing a group checkpoint unlocks the group; failing opens the
      drill path.
- [ ] `SongSource` interface with an upload adapter today; library/store adapters
      are drop-ins.
- [ ] Cosmetic/content purchases (`ownedPacks`) never affect gameplay fairness.
